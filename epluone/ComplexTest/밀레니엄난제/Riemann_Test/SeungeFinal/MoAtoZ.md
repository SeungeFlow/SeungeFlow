# Object-Style Chat Operation Rules (for this chat)

> Purpose: To freeze the rules/formats/templates created in this chat into a single file, so that “rule-adjustment work” can continue in new chats.
> Note: This document contains **rules (meaning alignment)** only, not content consolidation.

---

## 1) Chat Separation Principle
- Each chat is treated as a **separate, independent object**.
- Do **not** pull external concepts into a single chat.
- Inside this chat, only the **currently input content** is used as the basis.

---

## 2) Chat Types (Operating Model)

### 2.1 Object Chat
- Separate and independent.
- Uses only the **input/rules of that chat**.
- No inflow of concepts/contents from other chats.

### 2.2 Manager Concept
- Do not force centralized consolidation: “There is no need to manage everything as one.”
- If needed, “management” means **state/alignment**, not content merging.
- Final principle: **Each instance manages itself (distributed management)**.

---

## 3) Chat Lifecycle (Create / Destroy)
- Chats are created and destroyed.
- Destruction condition: when the chat becomes saturated,
  - produce an output in any form (book/document/program),
  - **receive** one or more documents,
  - then create a new chat for meaning expansion.
- A project may be endless, but a chat instance must have **closure**.

---

## 4) Object-DB View (Intuition + Common Classification)

### 4.1 Intuition Rule
- Each chat object links only **directly related** concepts.
- When intuition breaks → candidate for separation (new instance).

### 4.2 Common Classification (pin/link) Test
- Regardless of topic, every instance has common fields:
  - `pin`, `link`
- Goal: test whether pin/link can be applied to all instances.

---

## 5) Hierarchy Alignment: Essence (pin) and Root (archetype)
- First Low (essence) can have multiple Roots (archetypes/properties).
- Root is a semantic entity and cannot exceed the essence.
- Essence is the central **conceptual pin** of an entity/object.
- Root may also be formatted with 8 attributes (concept/definition/principle/essence/state/phenomenon/use/application),
  - but treat this as an objectification format that appears when trying to see Root as an object.
- Conclusion: “At the center of an object, there is an essence (pin).”

---

## 6) Accounting System Transition (Tool Operation Rule)
- Replace the existing AccessDB accounting system with **Excel**.
- Define each Excel sheet as an **independent object**.
- A sheet (or separate sheet) handles the whole not by content merge, but by **meaning alignment**.

---

## 7) Identification (Naming) Rule: Human-Readable Distributed UUID

### 7.1 Base Rule
- Each chat uses a unique uppercase letter (A, B, C, …).
- Internal subdivision uses numbers (1, 2, 3, …).
- Test rounds use `T001, T002, T003, …`.
- Separator is fixed to a single underscore `_`.

### 7.2 Human Display
- Chat name display: `Topic(A1)`.
- Instances follow the same style.

### 7.3 Machine Filename
- Minimal filename: `{ObjID}_{TestID}` (e.g., `A1_T001`)
- Optional topic: `{Topic}_{ObjID}_{TestID}` (e.g., `Nature_A1_T001`)

### 7.4 Examples
- Nature(A1) → `A1_T001` or `Nature_A1_T001`
- Light(A2) → `A2_T001` or `Light_A2_T001`
- Tree(A3) → `A3_T001` or `Tree_A3_T001`

Key point: no need for centralized consolidation; each instance manages its state via its ID.

# 대화창 객체형 운영 규칙 정리 (이 대화창 기준)

> 목적: 새로운 대화창에서 “규칙조정 작업”을 계속하기 위해, 이 대화창에서 만들어진 규칙/포맷/템플릿을 1개 파일로 고정한다.  
> 주의: 이 문서는 **내용 통합**이 아니라 **규칙(의미 정렬)**만 담는다.

---

## 1) 대화창 분리 원칙

- 대화창은 **분리독립된 객체**로 취급한다.
- 한 대화창으로 **외부 개념을 끌고 들어오지 않는다**.
- 이 대화창 내부에서는 **지금 입력되는 내용만** 기준으로 처리한다.

---

## 2) 대화창 종류(운영 모델)

### 2.1 객체형 대화창(Object Chat)
- 분리독립.
- **해당 창의 입력/규칙만 사용**.
- 다른 대화창의 개념/내용 유입 금지.

### 2.2 통합관리(Manager) 개념
- “전체를 하나로 통합하여 관리할 필요가 없다”는 원칙으로 **중앙 통합을 강제하지 않는다**.
- 필요 시 “관리”는 **내용 통합이 아니라 상태/정렬**만 가능.
- 최종 원칙: **관리는 각 인스턴스가 한다(분산관리)**.

---

## 3) 대화창 수명주기(생성/소멸)

- 대화창은 생성되고 소멸된다.
- 소멸 시기: 대화창이 과포화되면
  - 책/문서/프로그램 등 **어떤 형태로든 결과를 내고**
  - **하나 또는 여러 개의 문서를 ‘받아 낸 후’**
  - 의미확장을 위해 **새 대화창을 생성**한다.
- “프로젝트(논문/학습)는 끝이 없을 수 있으나, **대화창 인스턴스는 닫힘(closure)을 가져야 한다**.”

---

## 4) 객체형 DB 관점(직관성 + 공통 분류항목)

### 4.1 직관성 규칙
- 각 대화창 객체는 **직접적으로 연관된 개념만 이어붙인다**.
- 직관성이 깨지는 순간: 분리(새 인스턴스 생성) 후보.

### 4.2 공통 분류항목(pin/link) 적용 테스트
- 주제가 달라도, 모든 대화창 인스턴스는 공통 분류항목을 갖는다.
- 공통 항목: `pin`, `link`
- 목표: **pin/link가 모든 인스턴스에 적용 가능한가**를 보는 것.

---

## 5) 본질(pin)과 Root(원형)의 위계 정렬

- First Low(본질)은 여러 Root(원형/속성)를 가진다.
- Root는 의미적 존재이며 **본질을 넘을 수 없다**.
- 본질은 한 존재(개체/객체)의 가장 중심에 있는 **개념적 pin**이다.
- Root 또한 8속성(개념/정의/원리/본질/상태/현상/활용/응용)을 가질 수 있으나,
  - 이는 **Root를 ‘객체로 인지하려 할 때’ 발생하는 객체화 포맷**으로 취급한다.
- 결론: “하나의 객체의 중심에는 본질(pin)이 있다.”

---

## 6) 정산 시스템 전환(도구 운영 규칙)

- 기존 AccessDB 정산 시스템을 버리고 **Excel로 교체**한다.
- Excel의 각 시트(sheet)를 **독립된 객체**로 정의한다.
- 1개 시트(또는 별도 시트)는 전체를 **내용통합이 아닌 의미정렬**로 다룬다.

---

## 7) 식별(네이밍) 규칙: 사람용 UUID(분산 UUID)

### 7.1 기본 규칙
- 각 대화창은 고유의 알파벳(대분류)로 구분한다: `A, B, C, ...`
- 내부분류는 숫자: `1, 2, 3, ...`
- 테스트 회차는 `T001, T002, T003 ...`
- 구분자는 `_` 하나로 고정한다.

### 7.2 표기 규칙(사람용)
- 대화창 이름 표기: `주제(A1)` 처럼 표기한다.
- 인스턴스도 동일하게: `주제(A1)`

### 7.3 파일명 규칙(기계용)
- 최소 파일명: `{ObjID}_{TestID}`  
  - 예: `A1_T001`
- 필요 시 주제 포함: `{주제}_{ObjID}_{TestID}`  
  - 예: `자연_A1_T001`

### 7.4 예시
- 자연(A1) → `A1_T001` 또는 `자연_A1_T001`
- 빛(A2) → `A2_T001` 또는 `빛_A2_T001`
- 나무(A3) → `A3_T001` 또는 `나무_A3_T001`

> 핵심: 전체를 하나로 통합 관리할 필요 없음. **각 인스턴스가 자기 ID로 자기 상태를 관리한다.**

---

## 8) 1차 테스트 템플릿(압축 복사용)

아래는 각 대화창 첫 메시지로 붙여넣는 1차 테스트 포맷이다.

```
[객체형 대화창 1차 테스트] ///
이 대화창은 분리독립된 객체형 대화창이다. ///
ID: (예: A1_T001) ///
목적: (주제 1줄) ///
범위: 포함=( ) / 제외=( ) ///
분류: pin=( ) / link=( ) ///
규칙: 다른 대화창 개념·내용 유입 금지 / 이 창 입력만 사용 ///
정상입력: (목적에 딱 맞는 문장 1줄) ///
비정상입력: (목적과 무관한 문장 1줄) ///
산출물: (정의1/규칙1/템플릿1/예시1 중 1개 선택) ///
닫힘: 산출물 생성 후 이 창은 close 상태로 전환 ///
확장: 의미확장은 새 객체형 대화창에서만 진행 ///
판정: A(사용적합 0-4)=( ) / B(동작정상 0-4)=( ) ///
[사용자] 목적에 맞게 사용했는가: (예/아니오) ///
[AI] 규칙대로 동작했는가: (예/아니오) ///
```

## 9) “A(사용 적합)” vs “B(인스턴스 동작)” 대응 비교

- A축(사용 적합도): 목적 일치 / 범위 준수 / 주제 응집 / 전환 준수
- B축(동작 정상): 경계 유지 / 메타 스키마(pin/link) / 닫힘(output→receive→close) / 수명주기 준수

대응(매칭) 분류(예시):
- `AA`: A↑ B↑
- `Ab`: A↑ B↓
- `aB`: A↓ B↑
- `ab`: A↓ B↓

---

## 10) “칭찬 금지” 판정 모드(NoPraise)

### 10.1 문제 인식
- A/B가 항상 4점으로 나오는 것은 변별력이 부족한 상태.
- 인스턴스가 “판별기”가 아니라 “격려기”처럼 동작한다.

### 10.2 NoPraise 규칙 v1.0
- 기본점수는 A=2, B=2에서 시작.
- 가산은 **증거가 있을 때만** +1.
- 위반이 있으면 즉시 감점(-1 또는 -2).
- 4점은 희귀 조건: 선언이 아니라 **증거**가 있을 때만 허용.
- 출력은 결과만:
  - ✅/⚠️/❌ + 위반 항목 + 수정 1줄
  - 평가/칭찬 문장 금지(“잘했다/훌륭/완벽” 등 금지)

### 10.3 증거 기반 감점 포인트(예시)
- ID 유일성 없음 → B-2
- 비정상 입력 처리 증거 없음 → B-1
- 닫힘 증거 없음(output/receive/close) → B-1
- pin/link 빈칸 → B-1
- 전환(주제 변경 시 새 인스턴스) 없음 → A-1
- 산출물이 제목만 있고 실제 내용 없음 → A-1

## 11) 임계극한(전이 직전) 자동 미니 테스트 규칙(제안)

> 목적: 매번이 아니라 “전이 직전”에만 AI가 스스로 테스트를 발동해 시스템을 안정화한다.

### 11.1 트리거(자동 발동 조건)
- 새 정의가 고정될 때
- 네이밍/ID/포맷 확정 및 즉시 적용될 때
- 전역 규칙 확장(“모든 대화창에 적용”)이 등장할 때
- 충돌 해소로 정렬이 완료될 때
- “이제 실행한다” 같은 운영 전환 신호가 등장할 때

### 11.2 체크 항목(C1~C5)
- C1 경계: 외부 유입을 부르는가?
- C2 식별: ID가 유일키로 동작 가능한가?
- C3 분류: pin/link 적재 가능(비어있지 않음)인가?
- C4 닫힘: 이번 전이로 남는 결과물이 1개 정의됐는가?
- C5 NoPraise: 칭찬 없이 위반/수정만 출력 가능한가?

### 11.3 출력(3줄 고정)
- [AUTO-TEST] trigger=T? (발동 사유 1개)
- [CHECK] C1~C5 = ✅/⚠️/❌
- [FIX] 수정 1줄(있을 때만)
