# event_schema.md

> 문서번호: `Ctp24_EVENT_CONTEXT_0003`  
> 상태: Event / Context 적용 3회차  
> 필요한 모드: `Thinking 헤비`  
> 목적: Ctp24 구조원리와 구조연산기를 실제 Event / Context 구조로 작동시키기 위한 Event schema를 작성한다.  
> 위치 후보: `epluone/Event_Context/schema/event_schema.md`

---

## 0. 이 문서의 자리

이 문서는 Event / Context 적용의 세 번째 문서다.

1회차에서는 전체 구조식을 세웠다.

```text
Context_C + Event_C = C+1
```

2회차에서는 Context를 존재 형성장으로 정의했다.

```text
Context =
존재 형성장
```

이번 3회차에서는 Event를 정의한다.

Event는 단순 사건 설명이 아니다.

Event는 특정 조건과 시간흐름과 자리장 안에서 하나의 구조가 외부로 드러난 formed state다.

```text
Event =
formed structure
```

---

## 1. Event의 기본 정의

Event는 사건이 아니다.

또는 더 정확히 말하면, Event는 사건만이 아니다.

```text
Event ≠ incident only
Event ≠ chronological record only
Event ≠ result summary only
```

Event는 어떤 조건에서 어떤 구조가 formed 되어 관측 가능해진 상태다.

```text
Event =
formation output
+
structural singularity
+
observable formed state
```

따라서 Event는 다음을 포함할 수 있다.

```text
이론
사건
작품
기술
발명
문자체계
사회현상
전쟁
자본시장 구조
자연재해
정치적 전환
과학적 특이점
AI 구조
음악적 형성물
렌더링 구현물
```

그러나 이 목록이 Event의 본질은 아니다.

Event의 본질은 다음이다.

```text
어떤 구조가
어떤 조건 안에서
관측 가능한 formed state로 드러난 것
```

---

## 2. Event_C

Event_C는 Event가 하나의 C로 formed 된 상태다.

```text
Event_C =
event가 t와 p의 관계 속에서 formed 된 상태
```

Event_C는 단순 결과물이 아니다.

Event_C는 다음을 포함한다.

```text
formed structure
event condition
theory condition
technical condition
symbolic condition
historical position
structural output
diffusion path
```

즉 Event_C는 어떤 결과가 나왔다는 말이 아니라, 그 결과가 하나의 구조로 형성되었음을 뜻한다.

```text
Event_C =
구조 형성물
```

---

## 3. Event에서 m

Event에서 m은 중심에 놓인 formed structure다.

Context에서 m이 사람이나 존재 형성장에 가까웠다면, Event에서 m은 형성된 사건·이론·작품·기술·구조다.

```text
Event_m =
이론
사건
작품
기술
발명
문자체계
AI 구조
사회구조
상징체계
렌더링 결과물
음악적 형성물
```

그러나 Event_m도 단일하지 않다.

Event_m 역시 다중존재일 수 있다.

```text
Event_m =
하나의 formed structure
+
그 내부의 다중구조
```

예를 들어 하나의 이론은 단일 문장이 아니다.

그 안에는 전제, 기호, 정의, 관측기준, 수학적 구조, 오독 가능성, 적용 범위, 후대 영향이 있다.

하나의 작품도 단일 결과물이 아니다.

그 안에는 언어, 시대, 형식, 리듬, 상징, 독자 반응, 후대 해석이 있다.

따라서 Event_m도 내부 구조를 가진다.

---

## 4. Event에서 t

Event에서 t는 발생 시각만이 아니다.

```text
t ≠ date only
```

Event에서 t는 형성흐름이다.

```text
t =
형성 전 조건
+
누적
+
전환
+
임계
+
formed 순간
+
확산
+
후대 전이
```

Event의 t는 다음을 포함할 수 있다.

```text
선행 이론의 흐름
기술 발전의 흐름
사회 압력의 흐름
문제 축적의 흐름
작품 형성의 흐름
표현양식 변화의 흐름
발표 이후 확산 흐름
후대 재해석 흐름
```

연도와 날짜는 t를 표시할 수 있지만, t 그 자체가 아니다.

t는 Event가 formed 되기까지 누적된 시간적 압력과 전이의 흐름이다.

---

## 5. Event에서 p

Event에서 p는 발생 장소만이 아니다.

```text
p ≠ location only
```

Event에서 p는 formed structure가 놓인 자리장이다.

```text
p =
place
+
field
+
fabric
+
domain
+
condition field
```

Event의 p는 다음을 포함할 수 있다.

```text
학문장
기술장
정치장
사회장
언어장
문화장
문서장
실험장
관측장
기호장
AI computation field
```

Event는 항상 어떤 p 안에서 formed 된다.

p가 없으면 Event는 떠다니는 결과물이 된다.

---

## 6. Event에서 ?

Event에서 `?`는 Event를 무엇으로 볼 것인지를 정하는 관측기준이다.

```text
? =
6W1H
+
관측자의 시선
+
경계설정
+
해석조건
+
Event 판정 조건
```

Event에서 ?는 다음을 묻는다.

```text
누가 이 Event를 보는가?
무엇을 Event의 중심 m으로 둘 것인가?
언제부터 언제까지를 Event 형성과정으로 볼 것인가?
어디까지를 Event의 자리장 p로 볼 것인가?
왜 이 formed state를 Event로 볼 수 있는가?
어느 구조와 어느 조건 사이에서 Event가 발생했는가?
어떻게 Context와 연결되는가?
```

Event에서 `?`는 외부 질문표로만 있지 않다.

```text
? ∩ m 가능
? ∩ t 가능
? ∩ p 가능
? ∩ (m,t,p) 가능
```

예를 들어 어떤 이론의 정의 자체가 Event_m의 일부이면서 동시에 Event를 판단하는 기준이 될 수 있다.

어떤 작품의 형식 자체가 Event의 결과이면서 동시에 Event를 해석하는 관측기준이 될 수 있다.

---

## 7. Event는 결과가 아니라 형성상태다

Event를 결과로만 읽으면 평면화된다.

```text
Event ≠ result only
```

Event는 formed state다.

```text
formed state =
형성과정이 하나의 상태로 드러난 것
```

따라서 Event는 다음 구조를 가진다.

```text
선행조건
→ 누적
→ 압력
→ 임계사이영역
→ 극한임계전이
→ formed state
→ 확산
→ 후대 영향
```

Event는 formed 되는 순간만이 아니라, formed 되기 전의 조건과 formed 된 뒤의 확산을 함께 본다.

---

## 8. Event와 임계사이영역

Event는 임계사이영역을 가진다.

임계사이영역은 Event가 formed 되기 직전의 압력장이다.

```text
임계사이영역 =
Event가 Event로 드러나기 직전의 전이영역
```

이때 Event는 갑자기 생기지 않는다.

조건이 누적되고, 문제가 압축되고, 형식이 흔들리고, 기존 구조로는 더 이상 버틸 수 없을 때 Event가 formed 된다.

```text
누적
+
압력
+
경계 흔들림
+
전이
=
Event 형성조건
```

Event schema는 이 임계사이영역을 찾아야 한다.

---

## 9. Event와 극한임계전이

극한임계전이는 Event가 외부로 드러나는 전이 순간이다.

```text
극한임계전이 =
임계사이영역을 지나
Event가 formed state로 드러나는 순간
```

이 순간은 단순 날짜일 수 있지만, 날짜만으로 충분하지 않다.

예를 들어 논문 발표일, 작품 발표일, 발명일, 사건 발생일은 표식일 수 있다.

그러나 Event의 극한임계전이는 더 깊은 구조일 수 있다.

```text
발표일 =
표식

극한임계전이 =
구조가 이전 상태를 넘어 formed 되는 전이
```

Event schema는 표식과 전이를 구분해야 한다.

---

## 10. Event와 9dot0

Event는 9dot0 구조로 읽을 수 있다.

```text
9 dot 0
```

Event가 formed 되기 전에는 이전 구조의 끝점이 있다.

그 끝점은 단순 끝이 아니라 다음 formed state의 전제조건이 된다.

```text
9 =
이전 구조의 끝점

dot =
임계사이영역

0 =
Event의 formed 시작점
```

따라서 Event는 다음처럼 읽을 수 있다.

```text
이전 구조 9
→ dot
→ Event 0
```

여기서 dot은 단순 중간이 아니다.

dot은 Event가 formed 될 수 있는 무한한 잠재가능성의 장이다.

---

## 11. Event와 구조적 특이점

Event는 구조적 특이점을 포함할 수 있다.

그러나 특이점은 단순히 “중요한 일”이 아니다.

```text
특이점 ≠ 중요 사건 only
```

특이점은 구조가 이전 상태와 다른 방식으로 formed 되는 지점이다.

```text
구조적 특이점 =
이전 구조가 유지되다가
어떤 조건에서 다른 구조로 전이되는 지점
```

Event schema는 다음을 묻는다.

```text
이 Event는 무엇을 전환시켰는가?
이 Event 이전과 이후의 구조 차이는 무엇인가?
이 Event가 없었다면 어떤 Path가 열리지 않았는가?
이 Event는 어떤 C+1을 열었는가?
```

---

## 12. Event와 Context의 분리

Event는 Context와 분리된다.

그러나 고립되지 않는다.

```text
Context_C + Event_C = C+1
```

Context가 Event를 완전히 설명하는 것은 아니다.

Event가 Context의 단순 결과인 것도 아니다.

둘은 각각 하나의 C다.

```text
Context_C =
존재 형성장

Event_C =
구조 형성물
```

Path가 둘 사이의 relation을 연다.

```text
Path =
Context_C와 Event_C 사이의 relation operation
```

---

## 13. Event와 Core

Event는 Core seat에 닿는다.

Event_C는 하나 이상의 Core와 relation을 가진다.

예시:

```text
Event의 형성시간
→ Time-Flow Core

Event의 자리장
→ Place-Field Core

Event의 전이
→ 9dot0 Core

Event의 구조적 특이점
→ Structure-Body Formation Core

Event의 후대 확산
→ Path / C+1 Core
```

Event 작성은 Core를 채우는 작업이 아니다.

Event 작성은 Event가 어떤 Core seat에 닿는지 관측하는 작업이다.

```text
Event
→ Core seat 반응 관측
```

---

## 14. Event와 Path

Event는 Path 없이 Context와 연결되지 않는다.

Event_C가 있다고 해서 바로 C+1이 생기는 것이 아니다.

C+1은 Context_C와 Event_C 사이의 relation이 형성될 때 열린다.

```text
Context_C + Event_C = C+1
```

Path는 다음을 기록한다.

```text
Context_C는 무엇인가?
Event_C는 무엇인가?
둘 사이의 사이는 무엇인가?
둘 사이의 차이는 무엇인가?
어떤 전제조건이 있었는가?
어떤 임계사이영역이 있었는가?
어떤 C+1이 열렸는가?
```

---

## 15. Event 문서의 기본 구조

Event 문서는 다음 골격을 가진다.

```text
# Event_[target].md

## 0. 문서의 자리
## 1. Event의 중심 m
## 2. 형성 시간흐름 t
## 3. 형성 자리장 p
## 4. 관측기준 ?
## 5. 선행 조건
## 6. 누적과 압력
## 7. 임계사이영역
## 8. 극한임계전이
## 9. formed state
## 10. 구조적 특이점
## 11. 확산과 후대 영향
## 12. Core 반응 후보
## 13. Event_C 판정
## 14. Context 연결 후보
## 15. Path 후보
```

이 골격은 고정 양식이 아니다.

대상에 따라 조정될 수 있다.

그러나 첫 적용에서는 이 골격을 기준으로 삼는다.

---

## 16. Event 작성의 금지사항

Event 작성 시 금지사항:

```text
1. 사건 요약으로 만들지 않는다.
2. 결과물 설명으로만 닫지 않는다.
3. 날짜 나열로 만들지 않는다.
4. Event를 Context의 단순 결과로 낮추지 않는다.
5. Event와 Context를 섞지 않는다.
6. 발표일과 극한임계전이를 혼동하지 않는다.
7. 중요 사건이라는 이유만으로 특이점이라 하지 않는다.
8. 숫자패턴만으로 구조관계를 만들지 않는다.
9. Path 없이 C+1이 열렸다고 말하지 않는다.
10. Core를 content table처럼 채우지 않는다.
```

---

## 17. Event 작성의 허용사항

Event 작성 시 허용사항:

```text
1. Event_m을 formed structure로 읽는다.
2. t를 형성흐름과 전이로 읽는다.
3. p를 Event가 놓인 field로 읽는다.
4. ?를 관측자의 시선과 경계로 읽는다.
5. 선행조건과 누적 압력을 본다.
6. 임계사이영역과 극한임계전이를 구분한다.
7. formed state를 판정한다.
8. 구조적 특이점 여부를 따진다.
9. Core seat 반응 후보를 표시한다.
10. Path는 별도 문서에서 연다.
```

---

## 18. Event_C 판정 방식

Event_C는 마지막에 판정한다.

판정문은 다음 형식을 가진다.

```text
Event_C =
[m]이
[t]의 흐름과
[p]의 자리장과
[?]의 관측기준 안에서
formed 된 구조 형성물
```

예시 형식:

```text
Event_C =
상대성이론이
19세기 말-20세기 초 물리학의 관측위기와 수학적 도구의 전이 속에서
시공간 개념을 재구성하는 formed structure로 드러난 상태
```

이것은 예시다.

실제 대상에서는 source와 분석에 따라 다시 써야 한다.

---

## 19. Event와 숫자패턴 guard

Event 작성에서도 숫자패턴을 조심한다.

비슷한 숫자를 발견했다고 해서 관계로 확정하지 않는다.

```text
정수패턴 =
관측 후보

정수패턴 ≠ 관계 증거
```

같은 숫자라도 각 구조 안에서 의미가 다르다.

예를 들어 24가 여러 체계에서 보일 수 있지만, 숫자가 같다는 이유로 구조를 연결하면 오독이 발생한다.

Event schema는 숫자 유사성이 아니라 구조적 relation으로 Path를 열어야 한다.

---

## 20. Event schema의 핵심

Event schema의 핵심은 다음이다.

```text
Event는 사건 설명이 아니다.

Event는 m이 t와 p와 ? 안에서
formed structure로 드러난 상태다.
```

Event는 Context의 부록이 아니다.

Event는 Context와 동등한 C다.

```text
Context_C + Event_C = C+1
```

---

## 21. 닫힘

`event_schema.md`는 Ctp24 구조원리와 구조연산기를 실제 Event / Context 구조로 작동시키기 위한 Event schema로 작성되었다.

이 문서는 다음 판정으로 닫힌다.

```text
Event는 사건이 아니다.
Event는 formed structure다.

Event_C는
m이 t와 p와 ? 안에서 formed 된 구조 형성물이다.

Event는 Context와 분리되지만,
Path를 통해 Context와 relation을 맺어
C+1을 연다.
```
