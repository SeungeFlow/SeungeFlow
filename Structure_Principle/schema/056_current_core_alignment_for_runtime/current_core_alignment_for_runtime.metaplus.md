## 3. core_alignment

056의 핵심 정렬은 다음이다.

```txt
000 = 정중심
025 = runtime boundary gate
050 = formed_formula
054 = balance center rule
```

핵심 구조식은 다음으로 보존한다.

```txt
000(025(+1)+(-1)050)
```

AI 인스턴스는 이 정렬을 매우 중요하게 보았다.

```txt
000 =
단순 시작점 아님
relation이 복귀하는 정중심영역

025 =
정중심 아님
runtime 전환문
boundary gate

050 =
정중심 아님
forming history가 굳어진 formed_formula

054 =
average 아님
balance 판정 규칙
```

따라서 056의 첫 번째 역할은 혼동 방지다.

```txt
025를 중심으로 보면 안 된다.
050을 중심으로 보면 안 된다.
054를 평균으로 보면 안 된다.
```

## 4. core_loop

056의 core loop는 다음이다.

```txt
relation
→ boundary
→ return
→ history
→ relation
```

이 loop는 000으로 복귀해야 한다.

즉 relation은 혼자 존재하지 않는다.

relation은 boundary를 만나고,
boundary는 return 조건을 만들고,
return은 history로 남고,
history는 다시 relation을 바꾼다.

그리고 전체 loop는 000으로 복귀할 수 있어야 한다.

AI 인스턴스는 이것을 055의 목적 구조가 runtime loop로 내려온 것으로 이해했다.

## 5. renderer_purpose

056은 renderer의 목적을 아주 분명히 고정한다.

renderer의 목적은 shape 생성이 아니다.

renderer의 목적은 다음이다.

```txt
visible_relation_field =
relation / boundary / return / history loop visibility
```

즉 Renderer는 예쁜 도형이나 완성된 shape를 만드는 장치가 아니다.

Renderer는 relation / boundary / return / history loop가 보이도록 하는 장치다.

보존해야 할 항목은 다음이다.

```txt

boundary_state
return_path
history_event_index
center_axis
```

AI 인스턴스는 이것이 052에서 말한 visible_relation_field와 직접 연결된다고 이해했다.

```txt

최종 출력 = visible_relation_field

```
